# Termin Intermediate Representation (IR) Specification

**Version:** 0.1.0-draft
**Status:** Formative

---

## Purpose

The IR is the **handoff point** between the compiler and the runtime. The compiler produces it. The runtime consumes it. Without a clearly defined IR, the compiler and runtime fuse into one monolithic code-generation template — which is exactly the bug we're fixing.

The target architecture: the compiler emits an IR artifact (a JSON file conforming to this spec). The `termin-runtime` package loads that artifact and boots the subsystems. The generated application is the IR + an import of the runtime library. No more re-emitting eight subsystem implementations into every application.

---

# Part 1: Current IR (As-Implemented)

This section documents the IR as it exists in `ir.py` and is produced by `lower.py`. Claude Code: this is what you have. Part 2 is where you're going.

## Top-Level Structure: AppSpec

The IR root is `AppSpec`, a frozen dataclass containing all resolved application data. When serialized to JSON, it looks like:

```json
{
  "name": "string",
  "description": "string",
  "auth": AuthSpec,
  "tables": [Table, ...],
  "access_grants": [AccessGrant, ...],
  "state_machines": [StateMachineSpec, ...],
  "events": [EventSpec, ...],
  "routes": [RouteSpec, ...],
  "pages": [PageSpec, ...],
  "nav_items": [NavItemSpec, ...],
  "streams": [StreamSpec, ...],
  "computes": [ComputeSpec, ...],
  "channels": [ChannelSpec, ...],
  "boundaries": [BoundarySpec, ...],
  "error_handlers": [ErrorHandlerSpec, ...]
}
```

## Naming Convention: QualifiedName

Most named primitives use a `QualifiedName` triple:

```json
{
  "display": "stock levels",
  "snake": "stock_levels",
  "pascal": "StockLevels"
}
```

Generated by `_qname()` in `lower.py`. The `snake` form is canonical for internal references. The `display` form appears in UI. The `pascal` form is used for class/type names in generated code.

## Primitive Mapping

| Termin Primitive | IR Type(s) | Notes |
|---|---|---|
| Content | `Table` + `Column` | Content schemas become "tables" with "columns" — SQL-flavored naming |
| Identity | `AuthSpec` + `RoleSpec` | Flattened into auth block |
| State | `StateMachineSpec` + `TransitionSpec` | Attached to tables by `table` field |
| Events | `EventSpec` | Two modes: condition-based and JEXL-based |
| Compute | `ComputeSpec` + `ComputeParamSpec` | Five shapes, JEXL body lines |
| Channels | `ChannelSpec` | Intent-based direction/delivery |
| Boundaries | `BoundarySpec` + `BoundaryPropertySpec` | Contains tables and sub-boundaries |
| Presentation | `PageSpec` + many sub-types | The largest and most complex IR type |
| Access Control | `AccessGrant` | Denormalized from Content declarations |
| API | `RouteSpec` | HTTP routes with kind-based dispatch |
| Navigation | `NavItemSpec` | Flat list |
| Streaming | `StreamSpec` | Minimal — just description + path |
| Error Handling | `ErrorHandlerSpec` + `ErrorActionSpec` | Source-targeted with JEXL conditions |

## Key Types in Detail

### Column Types

```python
class ColumnType(Enum):
    TEXT = auto()      # strings, dates (SQLite), enums
    INTEGER = auto()   # whole numbers, booleans (0/1), foreign keys
    REAL = auto()      # currency, percentage, decimal numbers
    BOOLEAN = auto()   # (currently mapped to INTEGER for SQLite)
    DATE = auto()      # (currently mapped to TEXT for SQLite)
    TIMESTAMP = auto() # automatic fields, datetime
    JSON = auto()      # list types
```

### Compute Shapes

```python
class ComputeShape(Enum):
    TRANSFORM = auto()
    REDUCE = auto()
    EXPAND = auto()
    CORRELATE = auto()
    ROUTE = auto()
```

### Channel Intents

```python
class ChannelDirection(Enum):
    INBOUND = auto()
    OUTBOUND = auto()
    BIDIRECTIONAL = auto()
    INTERNAL = auto()

class ChannelDelivery(Enum):
    REALTIME = auto()
    RELIABLE = auto()
    BATCH = auto()
    AUTO = auto()
```

## Lowering Process

`lower.py` transforms the AST (`Program`) into the IR (`AppSpec`). Key behaviors:

- **Name resolution:** All names are resolved to snake_case for internal references. `_resolve_to_table()` handles singular/plural fuzzy matching.
- **Type mapping:** DSL types map to `ColumnType` via `_column_type()`. Several types collapse (e.g., boolean → INTEGER, date → TEXT for SQLite compatibility).
- **Access denormalization:** Access rules embedded in Content declarations are extracted into flat `AccessGrant` records.
- **State machine attachment:** State machines reference their Content by `table` (snake_case name). The `has_status_column` flag on Table indicates a state machine exists.
- **Event inference:** Events use fuzzy matching to resolve content names, including singular-to-plural and camelCase-to-snake_case conversions.
- **Route inference:** API endpoints are matched to Content via path prefix. Route kinds (LIST, GET_ONE, CREATE, UPDATE, DELETE, TRANSITION) are inferred from HTTP method + path shape. Transition target states use `VERB_STATE_MAP` for fuzzy resolution.
- **Page assembly:** Each UserStory becomes a PageSpec by walking its directives. Form fields are matched to Content schemas via `_best_content_for_fields()`. Aggregations use `_guess_content()` for fuzzy matching.
- **Badge hardcoding:** Navigation badges currently hardcode `"reorder_alerts"` — this is a known fragility.

## Known Issues in Current IR

1. **Content is called "tables" and fields are called "columns."** This leaks the SQL storage assumption into the IR. The IR should be storage-agnostic.
2. **PageSpec is a god object.** It has 22 fields covering tables, forms, filters, search, highlights, subscriptions, related data, aggregations, charts, static text, and expressions. Every new Presentation feature adds another field. This doesn't scale.
3. **Fuzzy matching everywhere.** `lower.py` uses regex-based name guessing, camelCase conversion, prefix matching, and a hardcoded `VERB_STATE_MAP`. These are fragile heuristics that will break on unexpected input.
4. **No Reflection representation.** The IR has no way to express Reflection queries or make runtime state available to expressions.
5. **State machines only attach to Content.** The runtime spec says State machines can attach to any primitive. The IR only supports `table` references.
6. **No error Channel wiring.** Error handlers have source references but no representation of the error Channel graph or escalation hierarchy.
7. **Type system is SQLite-flavored.** ColumnType values are storage types, not business types. `currency` and `number` both become `REAL`. `boolean` becomes `INTEGER`. This loses semantic information that the Presentation layer needs.
8. **No storage intent.** No field for `ephemeral`/`persistent`/`durable`/`auto` on Content.
9. **No IR version field.** No way to detect IR version mismatches between compiler and runtime.

# Part 2: Target IR (Future Vision)

This section defines what the IR should become. Changes are prioritized by how much they unblock the runtime library extraction.

## Structural Changes

### Add IR version

```json
{
  "ir_version": "0.2.0",
  "name": "...",
  ...
}
```

The runtime checks `ir_version` against its supported range and fails fast with a clear error if incompatible.

### Rename "tables" to "content" and "columns" to "fields"

The IR should use Termin vocabulary, not SQL vocabulary. This makes the IR storage-agnostic and consistent with the DSL and specs.

```json
"content": [
  {
    "name": { "display": "products", "snake": "products", "pascal": "Products" },
    "fields": [...],
    "has_state_machine": true,
    "initial_state": "draft",
    "storage_intent": "persistent"
  }
]
```

### Add storage intent

Each Content block gets a `storage_intent` field: `"ephemeral"`, `"persistent"`, `"durable"`, or `"auto"`. Default is `"auto"`. Can be overridden at Content level or inherited from application level.

### Split type system: business types vs storage types

The IR should carry **business types** (the semantic types from the DSL). The backend maps these to storage types. The Presentation renderer uses business types for formatting.

```json
{
  "name": "unit_cost",
  "display_name": "unit cost",
  "business_type": "currency",
  "required": true,
  "constraints": {
    "minimum": 0,
    "maximum": null,
    "unique": false,
    "min_length": null,
    "max_length": null,
    "pattern": null,
    "enum_values": []
  },
  "foreign_key": null,
  "is_auto": false,
  "list_element_type": null
}
```

Business types: `text`, `number`, `whole_number`, `currency`, `percent`, `boolean`, `date`, `time`, `datetime`, `automatic`, `list`, `enum`, `reference`.

The backend maps these to storage types internally. The IR never mentions SQL types, column types, or storage-specific representations.

### Generalize State machine attachment

State machines should reference any primitive, not just Content:

```json
"state_machines": [
  {
    "primitive_type": "content",
    "primitive_ref": "products",
    "machine_name": "product lifecycle",
    "initial_state": "draft",
    "states": ["draft", "active", "discontinued"],
    "transitions": [
      {
        "from_state": "draft",
        "to_state": "active",
        "condition": {
          "type": "scope",
          "scope": "write inventory"
        }
      },
      {
        "from_state": "active",
        "to_state": "discontinued",
        "condition": {
          "type": "jexl",
          "expression": "Identity.reflect.hasScope('admin inventory')"
        }
      }
    ]
  }
]
```

`primitive_type` is one of: `content`, `channel`, `boundary`, `compute`. `condition` supports both `scope` (simple scope check) and `jexl` (arbitrary expression).

### Add Reflection to IR

Reflection is a runtime capability, not an IR data structure. But the IR needs to indicate that Reflection is available and which primitives expose it. Add a top-level flag:

```json
{
  "ir_version": "0.2.0",
  "reflection_enabled": true,
  ...
}
```

The runtime, when it sees `reflection_enabled: true`, registers Reflection accessors for all primitives. JEXL expressions containing `.reflect.` are validated at compile time against the known accessor paths.

### Add error Channel wiring

Error handlers need to express the escalation hierarchy:

```json
"error_handlers": [
  {
    "source": "order_webhook",
    "source_type": "channel",
    "boundary": "order_processing",
    "condition_jexl": "error.kind == 'external'",
    "actions": [
      { "kind": "retry", "retry_count": 3, "retry_backoff": true, "retry_max_delay": "30s" },
      { "kind": "notify", "target_role": "order manager", "jexl_expr": "'Webhook failing: ' + error.message" },
      { "kind": "disable" }
    ],
    "log_level": "ERROR",
    "is_catch_all": false
  },
  {
    "source": "",
    "source_type": "",
    "boundary": "order_processing",
    "condition_jexl": null,
    "actions": [
      { "kind": "create", "target": "system_alert", "jexl_expr": "error" },
      { "kind": "escalate" }
    ],
    "log_level": "ERROR",
    "is_catch_all": true
  }
]
```

The `boundary` field indicates where in the hierarchy the handler lives. The TerminAtor uses this to determine escalation order.

## Presentation Overhaul (Preview)

The current `PageSpec` is a monolithic structure with hardcoded feature fields. This is the biggest structural problem in the IR and warrants its own specification document.

The future direction: replace `PageSpec` with a **component tree**. Each Presentation declaration becomes a tree of typed, composable UI components. The IR represents the tree; the renderer walks it.

Sketch of the target structure:

```json
{
  "name": "Inventory Dashboard",
  "slug": "inventory_dashboard",
  "role": "warehouse clerk",
  "required_scope": "read inventory",
  "root": {
    "type": "page",
    "props": { "title": "Inventory Dashboard" },
    "children": [
      {
        "type": "data_table",
        "props": {
          "source": "products",
          "columns": ["sku", "name", "category", "status"],
          "filters": [
            { "field": "category", "type": "enum" },
            { "field": "status", "type": "state" }
          ],
          "search_fields": ["sku", "name"],
          "highlight": { "condition": "[quantity <= reorderThreshold]" },
          "subscribe": "stock_levels"
        },
        "children": []
      },
      {
        "type": "aggregation",
        "props": {
          "label": "Total Stock Value",
          "expression": "[products.sum(.quantity * .unitCost)]"
        },
        "children": []
      },
      {
        "type": "chart",
        "props": {
          "source": "reorder_alerts",
          "period_days": 30,
          "chart_type": "line"
        },
        "children": []
      }
    ]
  }
}
```

This is composable, extensible (new component types don't require new PageSpec fields), and maps naturally to both the visual editor (each node is a rectangle) and the renderer (walk the tree, render each component).

This overhaul is deferred to a separate Presentation IR spec. The current PageSpec works for the existing example apps. The component tree is the path to arbitrary UI.

## Migration Path

The changes in Part 2 should be implemented incrementally:

**Phase 1 (immediate):** Add `ir_version`. Rename `tables` → `content`, `columns` → `fields`. Add `storage_intent`. This is a mechanical rename that unblocks the runtime library extraction.

**Phase 2 (soon):** Split business types from storage types. Generalize State machine attachment. Add error Channel wiring with boundary hierarchy. Add `reflection_enabled`. This unblocks the Reflection system and proper error handling.

**Phase 3 (later):** Presentation component tree. This is the big refactor that replaces PageSpec. It should happen after the runtime library is extracted and stable.

Each phase should include: update `ir.py` dataclasses, update `lower.py` lowering pass, update all `*_ir.json` test fixtures, update the backend to read the new structure, and update the linter/conformance tests.

## IR Serialization

The IR serializes to JSON for portability. The JSON representation must be:

- **Deterministic.** Same input always produces same output. Keys sorted, no random ordering.
- **Human-readable.** Indented with 2 spaces. No minification in development.
- **Versionable.** The `ir_version` field enables version negotiation between compiler and runtime.

The canonical file extension is `.termin.ir.json`. Example: `warehouse.termin` compiles to `warehouse.termin.ir.json`.

The runtime loads the IR with: `runtime = TerminRuntime.from_ir("warehouse.termin.ir.json")`.
